# Codex Project Context

## 项目定位

这是一个本地运行的闲鱼价差运营辅助后台。目标不是让用户手动添加和服务交付商品，而是让后台自动规划适合测试的无物流数字权益类商品，生成发布草稿，并在买家咨询或准备下单时辅助判断是否存在低价可采购货源。

当前产品目标：

- 优先选择不依赖人力服务交付的数字权益、兑换权益、小额点卡、代金券类候选。
- 接单前先确认可用范围、有效期、可转让性和当前低价货源。
- 接单后由用户人工付款采购低价来源，再交付给买家。
- 找不到低于采购上限的货源时，系统给出缺货/取消建议。

明确不做：

- 不自动绕过闲鱼登录、验证码、人脸、短信或风控。
- 不自动点击最终发布按钮。
- 不自动付款采购。
- 不做批量刷屏、伪装真人、规避平台检测。
- 不默认选择需要人力履约的服务类商品，例如 PPT 排版、简历优化、Notion 指导、Excel 问题整理。

## 技术结构

项目故意保持低依赖：

- 主程序：`app.py`
- 数据库：本地 SQLite，文件名 `xianyu_agent.db`
- 前端：`app.py` 内嵌 HTML/CSS/JS
- 运行方式：Python 标准库 `http.server`，无需 Node 或前端构建
- 默认地址：`http://127.0.0.1:8765`

运行：

```bash
python3 app.py
```

语法检查：

```bash
python3 -m py_compile app.py
```

## 核心数据表

`products`

商品池。保存商品名称、类目、关键词、成本、最低利润、风险缓冲、库存模式和备注。

`market_samples`

行情样本。保存从闲鱼页面、机会扫描、后台自动选品、履约搜索导入的标题和价格。

`drafts`

发布草稿。保存系统生成的标题、价格、描述、风险提醒和发布决策。

`orders`

履约判断记录。保存成交价、采购上限、履约状态、推荐货源和缺货建议。

`opportunities`

机会扫描结果。保存关键词、样本价格、建议价、预估利润和是否值得测试。

`publish_queue`

发布队列。后台自动选品后会把草稿放入这里，状态包括 `ready`、`filled`、`published`、`skipped`。

## 当前主要能力

### 后台自动选品

入口：左侧“后台自动选品”表单。

实现位置：

- 候选池：`AUTO_PRODUCT_CATALOG`
- 阻断词：`AUTO_BLOCKED_KEYWORDS`
- 处理函数：`run_autopilot`

当前候选池只应放入无需人力服务交付的商品，例如：

- 视频会员月卡兑换权益
- 音乐会员月卡兑换权益
- 网盘会员月卡兑换权益
- 咖啡代金券兑换权益
- 外卖代金券兑换权益
- 游戏点卡小额兑换权益

不要把“整理服务、咨询服务、设计服务、优化建议”等需要人工服务交付的商品放回候选池。

### 发布队列

自动选品会自动创建商品、行情样本、发布草稿，并加入 `publish_queue`。

发布队列提供“复制填表”按钮，生成一段 bookmarklet 脚本。用户进入闲鱼发布编辑页后运行脚本，系统尝试填写标题、价格和描述。

重要边界：脚本只填表，不点击最终发布。

### 行情采集辅助

入口：商品详情中的“打开闲鱼搜索”和“复制采样脚本”。

实现位置：

- `collector_script`
- `/collector.js?product_id=...`

采样脚本会读取当前浏览器页面可见文本，识别价格行，并回传到本地后台。它不处理登录、验证码或风控页面。

### 发布填表辅助

实现位置：

- `publisher_script`
- `/publisher.js?product_id=...`

脚本会根据当前商品实时生成标题、价格和描述，并尝试填入闲鱼发布页上的可编辑输入框。

### 机会扫描

入口：左侧“机会扫描”。

输入格式：

```text
会员月卡 | 15元 16元 18元 20元
某权益周卡 | 8 9.5 11 12
```

实现位置：

- `parse_opportunity_lines`
- `create_opportunities`
- `create_product_from_opportunity`

### 定价策略

实现位置：`analyze_market`

当前逻辑：

- 计算最低价、低价区间、中位数、去极值均价和高价区间。
- 根据竞争样本数和价格离散度决定加价。
- 建议价同时满足市场锚点、最低利润和风险缓冲。
- 采购上限 = 建议价或成交价 - 最低利润 - 风险缓冲。

### 订单履约增强

入口：商品详情中的“接单履约”。

用户输入成交价，并可粘贴最新低价货源。系统会先保存这些最新货源，再按采购上限筛选候选。

实现位置：`create_order`

如果没有候选低于采购上限，系统生成缺货/取消建议。

### 服务类旧数据清理

入口：左侧“清理服务类候选”。

实现位置：`cleanup_labor_products`

用途：移除旧版本自动生成的 `category = '无物流服务'` 商品及其发布队列、草稿、行情和订单。

## 合规与产品边界

开发时要持续保持这些边界：

- 文案要避免承诺“现货秒发”，因为默认是接单后采购。
- 文案应强调下单前确认可用范围、有效期、可转让性和当前可安排情况。
- 对票务、演唱会、电影票等高风险或强监管品类保持阻断或强提醒。
- 不要实现绕过平台风控、验证码、登录验证的功能。
- 不要实现自动付款或无人值守最终发布。
- 任何浏览器自动化都应保持“辅助填表/辅助采样/人工确认”的定位。

## 修改建议

如果继续开发，优先方向：

1. 给 `AUTO_PRODUCT_CATALOG` 增加更细的字段：适用平台、有效期类型、是否可转让、默认采购关键词。
2. 增加商品黑名单和类目风险等级，不只依赖关键词。
3. 发布队列增加“需要补图/需要确认类目/需要确认来源”等检查项。
4. 履约时增加候选来源可信度字段，例如卖家备注、历史失败次数、可核销说明。
5. 拆分 `app.py`，但只有在功能继续变大时再做，当前单文件便于低成本运行。
